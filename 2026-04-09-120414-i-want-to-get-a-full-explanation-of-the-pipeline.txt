в—Ҹ Here's the complete end-to-end pipeline:

  ---
  FitGuard Data Pipeline: Watch Sensors вҶ’ TFLite Classification

  Stage 1: Watch вҖ” Raw Sensor Collection

  wear/.../HealthTrackerManager.kt

  Uses the Samsung Health Tracking API to continuously sample at ~25 Hz:
  - PPG sensor: green, IR, red channels + timestamp
  - Accelerometer: x, y, z axes + timestamp

  Stage 2: Watch вҖ” Sequence Batching & Transmission

  wear/.../SensorSequenceManager.kt

  - Buffers PPG + accel samples into a 60-second sequence window
  - Splits into JSON batches (~75KB max each) and sends via Wearable Data API
  - Each batch includes metadata: sequence_id, batch_number, total_batches,
  activity_type
  - Path: /health_tracker_batch/{sequenceId}/{batchNumber}

  Stage 3: Phone вҖ” Data Reception

  mobile/.../WearableDataListenerService.kt

  - Receives batch data events, parses JSON into PpgSample(timestamp, green, ir,
   red) and AccelSample(timestamp, x, y, z)
  - Routes samples to the accumulator
  - Also logs raw data to JSONL files for persistence

  Stage 4: Phone вҖ” Batch Reassembly

  mobile/.../SequenceBatchAccumulator.kt

  - Tracks partial sequences in a map keyed by sequenceId
  - As batches arrive, appends samples and marks batches received
  - When all batches for a sequence arrive вҶ’ fires onSequenceReady(SequenceData)
   callback with sorted PPG + accel samples

  Stage 5: Phone вҖ” Windowing & Feature Extraction

  mobile/.../SequenceProcessor.kt

  Maintains sliding buffers and generates 60-second windows with 15-second step:

  PPG Processing (PpgProcessor.kt) вҶ’ 19 features:

  1. Bandpass filter (Butterworth 2nd-order, 0.5вҖ“5 Hz, forward-backward)
  2. Peak detection (rolling mean, beat timestamps)
  3. NN interval extraction (333вҖ“1500 ms valid range, Вұ30% outlier removal)

  Accel Processing (AccelProcessor.kt) вҶ’ 11 features:

  - Raw ADC вҶ’ m/sВІ conversion
  - Per-axis mean & variance (x, y, z) + magnitude mean, variance, peak
  - Step detection: bandpass 1вҖ“4 Hz, count half-cycles > 0.15 m/sВІ
  - Total steps + cadence (SPM)

  Total: 30 features extracted per window (19 PPG + 11 accel), but only the 19 
  PPG features go to the model.

  Stage 6: Phone вҖ” TFLite Model Inference

  mobile/.../FatigueDetector.kt + mobile/src/main/assets/fatigue_model.tflite

  Input preparation:

  1. Z-score normalize each of the 19 features using scaler_params.json
  (pre-computed mean/std per feature)
  2. Buffer into sliding window of 5 normalized feature vectors
  3. Only run inference every 3rd window (stride=3) after 5 warmup windows;
  intermediate windows are buffered via bufferWindowOnly()

  Model shape:

  Input:   [1, 5, 19]  вҶ’  batch=1, sequence_length=5, features=19
  Output0: [1, 2]      вҶ’  Current:  [pLow, pHigh]
  Output1: [1, 2]      вҶ’  Predict:  [futurePLow, futurePHigh]  (~60s ahead)

  - Output 0 ("Current"): probability of low vs high fatigue for the current
  moment
  - Output 1 ("Predict"): probability of low vs high fatigue ~60 seconds in the
  future

  Classification from pHigh:

  Stage 7: Phone вҖ” Smoothing & Alerting

  mobile/.../FatigueAlertManager.kt

  - EMA smoothing: smoothed = 0.4 Г— raw + 0.6 Г— previous (reduces noise)
  - Trend analysis: linear slope over last 5 smoothed values вҶ’ INCREASING /
  STABLE / DECREASING
  - Alert triggers: level escalation, or 3+ consecutive pHigh вүҘ 0.50, with 60s
  cooldown
  - Sends alerts to both phone notification (with vibration patterns per level)
  and watch via /fitguard/fatigue/alert
