     Fix Prediction Instability: Stride Control + EMA Smoothing

     Context

     When a 60s batch arrives from the watch, SequenceProcessor generates ~4
     sliding windows (60s window, 15s step) and calls
     FatigueAlertManager.onFeatureWindow() for each. This fires 4 predictions
     instantly, causing fatigue levels to flip wildly. The model was trained
     with step=3 (every 3rd window), but deployment runs inference on every
     window (stride=1).

     Goal: Match deployment inference cadence to training (stride=3), add EMA
     smoothing to prevent single-prediction spikes, and require sustained high
     fatigue before firing alerts.

     ---
     Part 1: Add bufferWindowOnly() to FatigueDetector

     File: mobile/src/main/java/com/example/fitguard/data/processing/FatigueDete
     ctor.kt

     Add a new method that normalizes and buffers features WITHOUT running
     inference:

     fun bufferWindowOnly(rawFeatures: FloatArray) {
         val normalized = normalize(rawFeatures)
         windowBuffer.addLast(normalized)
         if (windowBuffer.size > SEQ_LENGTH) windowBuffer.removeFirst()
     }

     This keeps the detector's buffer in sync on non-prediction windows so that
     when inference does run, it has the latest 5 windows.

     ---
     Part 2: Add bufferWindowOnly() + EMA Smoothing + Consecutive-High to 
     FatigueAlertManager

     File: mobile/src/main/java/com/example/fitguard/features/fatigue/FatigueAle
     rtManager.kt

     New fields:

     - smoothedPHigh: Float = -1f (EMA accumulator, -1 = uninitialized)
     - lastRawPHigh: Float = 0f
     - consecutiveHighCount: Int = 0
     - EMA_ALPHA = 0.4f
     - CONSECUTIVE_HIGH_THRESHOLD = 3 (require 3+ consecutive predictions above
     0.50)

     New method bufferWindowOnly(context, features):

     - Call ensureInitialized(context)
     - Call detector.bufferWindowOnly(features) (buffer only, no inference)

     Modify onFeatureWindow():

     After detector.addWindowAndPredict() returns a result:
     1. Store lastRawPHigh = result.pHigh
     2. Compute EMA: if smoothedPHigh < 0 then smoothedPHigh = result.pHigh else
      smoothedPHigh = alpha * result.pHigh + (1-alpha) * smoothedPHigh
     3. Re-classify using smoothedPHigh for the 4-level mapping (same
     thresholds: 0.25, 0.50, 0.75)
     4. Track consecutiveHighCount: increment if smoothedPHigh >= 0.50, reset to
      0 otherwise
     5. Only fire alert (notification + send to watch) when consecutiveHighCount
      >= CONSECUTIVE_HIGH_THRESHOLD AND level escalated
     6. Expose lastRawPHigh and smoothedPHigh as readable properties (already
     object-level)

     Modify reset():

     Also clear smoothedPHigh = -1f, lastRawPHigh = 0f, consecutiveHighCount = 0

     Part 3: Stride Control in SequenceProcessor

     File: mobile/src/main/java/com/example/fitguard/data/processing/SequencePro
     cessor.kt

     - windowsSinceLastPrediction: Int = 0

     Modify broadcastResult():

     private fun broadcastResult(fv: FeatureVector) {
         // ... existing broadcast intent (unchanged ‚ÄĒ FatigueViewModel still
